## Codebase Patterns

- **Cache singleton pattern** (`app/cache/redis_client.py`): lazy thread-safe `get_redis()` that resolves URL from `Config.REDIS_URL` with env-var fallback. Uses `ConnectionPool.from_url(max_connections=50, decode_responses=False)`. Future repos should import `from app.cache import get_redis` and serialize payloads explicitly (JSON/msgpack), not rely on `decode_responses`.
- **Health endpoint pattern** (`app/routes/health.py`): separate blueprint for cheap anonymous probes; exceptions from Redis/DB checks are caught and surfaced in the JSON body, not re-raised. Returns 200/503 based on overall status. `/api/stats` remains the ECS container healthcheck target — don't repoint it until a story says so.
- **CDK security-group ordering**: for resources like ElastiCache that need SG ingress from ECS services created later in the stack, define the resource's own SG up front (outbound disabled), then call `sg.add_ingress_rule(peer=service.connections.security_groups[0], ...)` after each ECS service is constructed. Avoids circular construct ordering that would happen with `allow_from`.
- **Prod vs dev compose**: `docker-compose.yml` is the canonical service graph; `docker-compose.override.yml` is the dev-only layer (SQLite, exposed ports, profile-disabled prod-only services). Add `depends_on.<svc>.condition: service_healthy` on the app service so Flask waits for deps to be healthy before starting.
- **Alembic dialect gate for Postgres-specific DDL** (`migrations/versions/014_add_companies_table.py`): when a migration needs a Postgres-only feature (partial index, `JSONB`, `ARRAY`, tsvector, etc.) but the project also runs on SQLite locally, branch on `op.get_bind().dialect.name == "postgresql"` and emit a portable fallback in the `else`. Example: `op.create_index(..., postgresql_where=sa.text("..."))` in Postgres, plain `op.create_index(name, table, [col])` on SQLite. Keeps one migration file + makes local test bootstraps survive.
- **ORM vs Pydantic namespace split** (`app/models/`): `app/models/__init__.py` holds SQLAlchemy ORM classes (legacy compatibility — all existing `from app.models import User` imports resolve here). Domain-specific Pydantic schemas live as sibling modules `app/models/<domain>.py` (e.g. `app/models/company.py`) and are imported explicitly as `from app.models.company import Company`. Do NOT re-export Pydantic classes from `__init__.py` — keeps the two namespaces separate so ORM `Company` and Pydantic `Company` can share a name without shadowing.
- **Package conversion preserving history**: when converting `pkg.py` → `pkg/__init__.py`, use `git mv pkg.py pkg/__init__.py` (after `mkdir -p pkg`). Git records it as a rename, history follows.
- **SCD-2 via Postgres trigger** (`migrations/versions/016_add_company_fundamentals.py`): current-view table + append-only history table with a BEFORE UPDATE trigger that copies OLD row into history and updates NEW.updated_at := NOW(). Trigger + underlying PL/pgSQL function share the same name (spec convention — the pg_trigger assertion keys on `tgname`). Downgrade order is load-bearing: drop trigger → drop function → drop tables (Postgres refuses to drop a function whose triggers still exist). Dialect-gated — SQLite cannot express this cleanly, so the corresponding repo writes history rows manually on non-Postgres dialects. Trigger body uses `CREATE OR REPLACE FUNCTION` so re-applying the migration after a downgrade works without errors.
- **DESC-order indexes via sa.text()** (migrations 015–019): Alembic's `op.create_index(name, table, [col])` cannot express the DESC modifier directly, but passing `[sa.text("ticker"), sa.text("period_end DESC")]` works dialect-agnostically — Postgres honors DESC, SQLite tolerates it. Use this pattern for every index that supports "latest N" read paths (financials, fundamentals_history, institutional_holders, insider_transactions).
- **Dedup by PK vs UNIQUE vs composite UNIQUE** (Layer 2+3 tables): three patterns in use. (a) **Composite PK dedup** — `company_financials` uses PK (ticker, period_end, period_type) so re-ingesting the same filing fails with a PK violation, no app logic needed. (b) **UNIQUE dedup with surrogate BIGSERIAL id** — `institutional_holders` keeps id as the natural monotonic PK for event ordering and overlays a UNIQUE on (ticker, institution_cik, report_date) as the dedup gate. (c) **Idempotent-ingest UNIQUE beyond spec** — `insider_transactions` (US-007) uses a surrogate BIGSERIAL PK plus a composite UNIQUE on (ticker, insider_name, transaction_date, transaction_type, shares, form_type) that deliberately EXCLUDES price_per_share and total_value. Rationale: SEC and redistributors publish prices to slightly different precision, so including price in the key would duplicate the same Form 4 when ingested from multiple sources. Pick the strictest pattern that lets re-ingest be a no-op from all plausible upstream paths.
- **FK ON DELETE RESTRICT for reference tables** (every Layer 2/3 table → companies): spec mandates RESTRICT (not CASCADE) so master-row deletion is explicitly blocked while child rows exist. Companies that delist should set `delisted_at` instead of being deleted — the Alembic migrations enforce this with `sa.ForeignKeyConstraint([...], [...], ondelete='RESTRICT')`. Declare ORM-side `ForeignKeyConstraint` in `__table_args__` rather than inline `ForeignKey()` on the Column so the RESTRICT is visible at the table level.
- **BigInteger autoincrement → BIGSERIAL** (migrations 018/019): `sa.Column('id', sa.BigInteger(), primary_key=True, autoincrement=True)` emits `BIGSERIAL` on Postgres (shows as `bigint DEFAULT nextval(...)` in `\\d`) and `INTEGER AUTOINCREMENT` on SQLite. Portable across dialects with no additional code.

- **Pydantic model JSON round-trip** (`app/repositories/base.py`): `cached_get` uses `model_validate_json(raw_bytes_or_str)` for parse (Pydantic v2 accepts both); `model_dump_json()` for serialization. For lists, `json.dumps([x.model_dump(mode='json') for x in items])` over the whole collection so one Redis GET recovers everything — the list-path doesn't try to cache individual items separately.
- **Cache-aside tolerance policy** (spec §2 + `BaseRepository`): every Redis call is wrapped in `try/except Exception` that logs a warning via `logging.getLogger("signal.repo")` and falls through. GET failure → run the DB loader. SETEX failure → swallow (the DB value is already loaded; next reader will try again). DELETE failure → swallow. This is a conscious correctness/latency tradeoff — Redis is an accelerator, not the system of record.
- **Pattern-based invalidation via SCAN** (`BaseRepository.invalidate_pattern`): `redis.Redis.scan_iter(match=pattern, count=200)` walks the keyspace non-blockingly and deletes matches. Count returned is best-effort (on error, logs and returns the partial count). Used by competitors (`company:{TICKER}:competitors:top*`), institutions (`company:{TICKER}:institutions:top*`), and insiders (`company:{TICKER}:insiders:*d`) — scenarios where the exact cached key suffix (top-N for different N, last-N-days variants) is not known at write time.
- **Sticky-negative avoidance** (`BaseRepository.cached_get`): a `None` return from `db_loader` is NOT written back to Redis. An absent row shouldn't poison the cache for 24h+; let the next reader retry the DB. The list variant DOES cache an empty list — that's a legitimate "there are no competitors for AAPL" answer.
- **Integration-test marker convention** (`pyproject.toml`): register `integration` pytest marker + gate tests on `os.environ.get("INTEGRATION_TEST")`. Unit tests are always fast and mocked; the integration-marked subset exercises the real Postgres trigger / ON CONFLICT DO UPDATE path / SCAN invalidation against live services. Tester runs the full suite with `INTEGRATION_TEST=1 DATABASE_URL=... REDIS_URL=...`; unit-only runs drop the env and see the test skipped.
- **ON CONFLICT DO UPDATE for idempotent ingest** (`financials_repo.append`, `institutions_repo.append_batch`): dialect-gated. On Postgres, use `from sqlalchemy.dialects.postgresql import insert as pg_insert` against the mapped class (`pg_insert(FinancialsORM)` — passing the class directly is the mypy-friendly idiom; `__table__` triggers typing mismatches). Build `update_cols = {c: stmt.excluded[c] for c in payload if c not in PK_COLS}` and call `stmt.on_conflict_do_update(constraint=..., set_=update_cols)`. On SQLite (test bootstrap), wrap `session.add(row); session.commit()` in `try/except IntegrityError` and run a targeted UPDATE on rollback — same semantics, portable.
- **SCD-2 dialect emulation** (`fundamentals_repo.upsert`): the Postgres trigger `fn_snapshot_fundamentals_before_update` owns history writes on production. For SQLite test runs the trigger doesn't exist, so the repo manually appends a `CompanyFundamentalsHistory` row with `valid_from=old.updated_at`, `valid_to=now()` before mutating the current row. Guarded by `session.bind.dialect.name != "postgresql"` — the real deploy path never runs this branch. When touching this file, use `setattr(row, "updated_at", ...)` for the `updated_at` refresh; direct assignment to legacy-Column attributes tripping mypy as `Column[datetime]` vs `datetime` mismatch.
- **Ticker normalization at the repo boundary**: every public repo method uppercases the ticker argument immediately and reuses the upper variant for the cache key, the SQL filter, and the Pydantic payload. This keeps the `company:{TICKER}:...` namespace deterministic regardless of caller casing — callers can pass `"aapl"` and `"AAPL"` interchangeably.

- **ThreadPoolExecutor fan-out for service aggregators** (`app/services/company_service.py`): synchronous Flask + sync SQLAlchemy doesn't preclude parallelism for I/O-bound workloads. The service owns a `with ThreadPoolExecutor(max_workers=6) as ex:` block, submits one future per domain (`ex.submit(self._fetch_with_backfill, ticker, domain, loader, backfill)`), and collects results in a per-name dict. Each future is wrapped in try/except so a single domain failure flips `partial=True` instead of raising. Same signatures survive a future async migration — swap `ex.submit` for `asyncio.gather` and the call sites don't move. Measured locally: six 100ms repo reads complete in <120ms total.

- **Redis SETNX per-(ticker, domain) mutex pattern** (`app/services/company_service.py._fetch_with_backfill`): for on-demand backfill that must not thunder-herd the upstream API, acquire `lock:company:{TICKER}:{domain}` via `r.set(key, b"1", nx=True, ex=30)`. If SETNX succeeds, we own the fetch: run `backfill(ticker)` which hits Polygon/EDGAR and persists via the repo's upsert/append. If it fails (another worker holds the lock), poll the SAME repo at 200ms intervals up to 5s deadline — the holder's `repo.upsert_batch` / `repo.append` populates Redis via invalidate-then-populate, so a reader polling the repo picks up the persisted result naturally. The lock is always released in `finally`, and a Redis outage during acquire falls through to upstream-direct (swallow + log — Redis is an accelerator, not a source of truth). 30s TTL is a belt-and-braces timeout in case the fetch crashes before `delete`. Verified with 5 concurrent threads on a cold key → exactly 1 upstream call. Testable without live Redis by patching `app.services.company_service.get_redis` with a small `_FakeRedisMutex` that implements SETNX on an internal dict.

- **Dual-write soak pattern** (`app/routes/market.py`, introduced in US-013): when migrating from a legacy cache/table to a normalized one, keep both writers active during a 7-day soak. Recipe: (a) add a `.. deprecated:: US-XXX` Sphinx directive + `warnings.warn(DeprecationWarning, stacklevel=2)` on the legacy class's `__init__` so callers see a pytest/runtime warning but nothing breaks; (b) leave the legacy read+write path unchanged — response shape stays byte-identical for frontend compat; (c) on every successful upstream fetch, invoke a best-effort `_persist_<domain>()` helper that maps the upstream dict → Pydantic model and writes to the new repo. Every `_persist_*` function is wrapped in `try/except Exception: logger.debug(...)` so dual-write failures never bleed into the HTTP response; (d) for FK-RESTRICT tables (anything downstream of `companies`), the persist helper first runs a one-shot `CompanyRepository.get(ticker)` check and bootstraps the master row via `_persist_company_master` if absent — otherwise the FK blows up on a cold ticker. After the soak, US-018 removes the legacy L2 + the deprecated class; US-014 does the bulk historical backfill. Verified: all 14 pre-existing market_routes tests pass unmodified, proving response shape is preserved.

- **APScheduler job registration in worker.py** (introduced in US-015 / US-016): the worker process exposes a module-level `scheduler = _build_scheduler()` so tests and assertions can introspect job IDs WITHOUT calling `scheduler.start()` (which spawns a background thread that races test setup). `_build_scheduler()` MUST be defined AFTER the `run_*` wrapper functions because APScheduler's `add_job()` resolves the callable eagerly at registration time — passing a string ref or function object both trigger an immediate import lookup. Each registered job uses `replace_existing=True` so re-imports inside the same interpreter (gunicorn worker reloads, tests) don't trip `ConflictingIdError`. The wrapper pattern is `def run_X(): tickers = _active_tickers(); return _run_with_audit("X", ingest_X, tickers)`, where `_run_with_audit` writes a `AuditLog(action='ingestion_failure', admin_email='system')` row on exception and never raises. `main()` calls `scheduler.start()` AFTER the feed-refresh sync loop is set up; `scheduler.shutdown(wait=False)` runs in the SIGTERM handler.

- **13F window-gating pattern** (`app/ingestion/edgar_calendar.py`, US-015): `get_active_13f_window(now)` is a pure function that returns `(date, date) | None`. Walks 5 candidate `(year, quarter)` tuples — `(now.year-1, "Q4")` plus `(now.year, "Q[1-4]")` — and for each one reads a Redis override at `edgar:13f:deadline:{YYYY}-Q{n}` before falling back to the hardcoded date `(Q4→Feb 14 of next year, Q1→May 15, Q2→Aug 14, Q3→Nov 14)`. Returns the first window `[deadline-14d, deadline+1d]` that contains `today`. The hourly `edgar_13f_intensive` job calls this and no-ops when `None`; ~96% of the year is a no-op. The monthly `edgar_13f_calendar_probe` job persists better-known deadlines into Redis. Redis read failures degrade gracefully to the hardcoded path — calendar logic NEVER raises into the scheduler. Pattern reused for any future periodic-burst-around-fixed-deadlines ingest (10-K annual filings, etc.).

- **Ingester → repo → invalidate flow** (`app/ingestion/edgar_ingester.py`, `app/ingestion/market_data_ingester.py`, US-015 / US-016): every ingest function is the same 4-step skeleton. (1) Loop over `tickers`. (2) Fetch via the existing client singleton (rate limit + L1/L2 cache preserved — DO NOT instantiate new EdgarClient/PolygonClient). (3) Map the upstream dict → Pydantic model with permissive `Optional[T]` handling and per-ticker `try/except` so one bad row never aborts the batch. (4) Call the repo's idempotent write helper (`upsert` / `append` / `upsert_batch` / `append_batch`) — the repo handles Redis invalidation on its end via `invalidate(key)` or `invalidate_pattern(pattern)`. Bootstrap the `companies` master row via `_ensure_master(ticker)` BEFORE any Layer 2/3 write — see "Backfill destination-ticker bootstrap pattern" below. Functions return `dict[str, int]` (ticker → rows written) so the scheduler wrapper can log a single line per job tick.

- **Backfill destination-ticker bootstrap pattern** (introduced in US-014 to fix the FK-RESTRICT violation flagged by US-013): whenever a writer touches a Layer 2/3 table whose row references a master ticker that may not exist in `companies` (e.g. `company_competitors.competitor_ticker` FK with `ondelete=RESTRICT`), the writer MUST bootstrap that destination row first: `if company_repo.get(t) is None: company_repo.upsert(Company(ticker=t, name=t))`. Done in 4 places: `scripts/backfill_company_data.py._process_competitors`, `app/services/company_service.py._backfill_competitors` (fixed via this pattern), `app/routes/market.py._persist_competitors` (also fixed), `app/ingestion/edgar_ingester.py._ensure_master`. A future cleanup is to push this bootstrap into `CompetitorsRepository.upsert_batch` itself so callers don't need to remember; same for any other repo whose write target has multi-end FKs.

- **TokenBucket rate limiter for upstream-API fan-out** (`scripts/seed_sp500.py`, US-017): when a script makes many calls to an external API with a documented per-second rate limit, wrap each call site with a shared `TokenBucket(rate_per_sec).acquire()`. Pattern: thread-safe class with `rate`, `capacity` (set to `1.0` so the burst rate equals the long-run rate — strict adherence from the very first call), `tokens` initialised to capacity, `last` monotonic timestamp, and a `lock`. `acquire` reads `now`, regenerates tokens by `elapsed * rate`, clamps at capacity, then either decrements (token available) or sleeps `(1.0 - tokens) / rate` and consumes (token short). One bucket per upstream — the seed script keeps `edgar_bucket = TokenBucket(8.0)` and `polygon_bucket = TokenBucket(4.0)` at module level. Each ingester invocation is wrapped at the call site (`bucket.acquire(); ingester(...)`); the ingester's internal client throttles act as a second-line defence. Validated with `10 acquires at 5/sec → 1.80s` (>1.4s rate-limit assertion). The `BURST_CAPACITY=1.0` choice diverges from the textbook "capacity = rate" — for upstream limits like SEC's 10 req/sec, a burst of 10 immediate calls would briefly exceed the gate; capacity-of-1 keeps us strictly under from the first call.

- **JSON-progress-file resumability for one-shot scripts** (`scripts/seed_sp500.py`, US-017): for any long-running script that processes a list of work items and might be interrupted (network failures, OOM, operator Ctrl-C), persist progress to a single JSON file with `completed: list[str]`, `failed: list[dict]`, `partial: list[dict]`, `started_at`, `last_updated`. On startup, load `completed` into a set and skip those items from the input list — log `skipped_already_done=N` at the start. After EACH item completes (success / partial / failure), flush the JSON via tempfile + `os.replace` for atomic-write semantics (a half-written file from a crash will not be loaded). Provide three flags: `--reset` (delete the progress file), `--limit N` (process only first N items, useful for smoke tests), `--dry-run` (plan-only, zero side effects). Partial outcomes (some sub-steps failed but at least one succeeded) belong on the `partial` list AND on `completed` so re-runs without `--reset` don't endlessly retry — operators use `--reset` for a forced clean pass. Final-summary line plus an audit-log row (`event_type='<script>_complete'`) gives observability without tail -f. Pattern is reusable for any future bulk ingest / migration script.

**Learnings:**
- **Always FLUSHDB between adversarial runs against shared Redis.** The repo layer's cache-aside semantics deliberately cache `Company` rows with 24h TTL (per §10/§44 of progress.txt) — re-running tests against the same Redis with a wiped Postgres makes `company_repo.get(t)` lie about table state. Prefer either a per-test Redis db (REDIS_URL=redis://127.0.0.1:6379/N for some test-scoped N) or an explicit FLUSHDB in test setup. The wider implication: any future test that wipes companies but not Redis will see ghost rows for any ticker the implementor previously ran through. Add to Codebase Patterns.
- **SCD-2 triggers on identical-value UPDATEs require explicit dirty-bit forcing.** SQLAlchemy elides the UPDATE when no field changed. To exercise the trigger in a test, either (a) `setattr(row, 'updated_at', datetime.utcnow())` to force a dirty flag, or (b) deliberately perturb a value. Worth a code comment in fundamentals_repo.upsert to flag this for future debuggers.
- **APScheduler add_job string-ref vs callable-ref are NOT equivalent for module-load-time imports.** Implementor learned this in §317; tester confirmed by introspecting `scheduler.get_jobs()` without `.start()` — all 8 jobs (5 EDGAR + 3 Polygon) are present with stable IDs at module import time. `polygon_fundamentals` alias job exists exactly because the spec asserts that ID directly; without the alias the cron-scoped market/off-hours jobs would leave the spec assertion brittle to scheduling-grammar refactors.

**Learnings:**
- **TokenBucket capacity choice matters for the rate-limit assertion.** Initial implementation set `capacity=rate`, which let the first `rate` calls fire instantly — so 10 acquires at 5/sec took 1.0s (5 free + 5 paced), failing the prompt's `>=1.4s` assertion. Setting `BURST_CAPACITY=1.0` enforces strict per-call spacing from the first call: 10 acquires at 5/sec → 1.80s. The tradeoff: a strict capacity-of-1 forfeits the natural smoothing benefit of a token bucket, but for upstream APIs with hard per-second gates that is exactly the right tradeoff (a burst of 8 immediate EDGAR calls would briefly exceed the 8 req/sec ceiling; capacity-of-1 never does).
- **Env order is load-bearing for `Config.DATABASE_URL`.** `Config.DATABASE_URL` is resolved at class-definition (import) time via `_build_database_url()` which reads `os.environ['DATABASE_URL']`. So whether the seed script writes to Postgres or SQLite depends entirely on the env shape at the moment Python imports `app.config`. The first run of self-validation hit this — `export ... ; source .env ; python ...` got SQLite because `.env` overwrote the export. Documented in the Tester notes above.
- **Partial-vs-completed bookkeeping is a real product decision.** Adding partial tickers to `completed` favours forward progress over completeness; the alternative (keep partial out of `completed` and retry forever) leads to an infinite-loop on tickers that legitimately have no Form 4 history (e.g. recently IPO'd). The chosen semantics matches the spec's "Resumable — re-running after a crash skips already-completed tickers" by treating partial-success as a kind of completion.
- **Audit log fallback to structured logging.** When `AuditLog` insert fails (e.g. SQLite test path without the table, or a transient DB error) we fall through to `logger.info('sp500_seed_complete %s', json.dumps(...))`. The event is still observable in the worker log even if the DB row is missing — this is the same pattern as `_audit_failure` in `app/worker.py`.
