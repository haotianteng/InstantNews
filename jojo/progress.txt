### Stdout-capture test pattern (EMF / structured logs)
For modules that emit structured log lines via `print(..., flush=True)`, use pytest's `capfd` fixture (captures at the file-descriptor level, so it sees `flush=True` output correctly). Parse the captured string by splitting on newlines and ignoring blanks:

```python
def _parse_single_line(captured: str) -> dict:
    lines = [ln for ln in captured.splitlines() if ln.strip()]
    assert len(lines) == 1, f"expected 1 line, got {len(lines)}"
    return json.loads(lines[0])

def test_emit(capfd):
    emit_metric("NS", "M", 1.0, dimensions={"A": "b"})
    out, _err = capfd.readouterr()
    d = _parse_single_line(out)
    ...
```

`capfd.readouterr()` clears the buffer between calls, so multiple emissions in one test work if you read after each.

### EMF JSON shape (from live spec fetch 2026-04-17)
- `Dimensions` is a list of lists. Outer array has `minItems: 1` (spec-mandated) ŌĆö even when emitting "no dimensions", we must render `Dimensions: [[]]` (one empty DimensionSet). This is valid because each DimensionSet has `minItems: 0`.
- Dimension *values* must be strings at the root level. Ints/floats/bools must be coerced (`str(value)`) before emission or CloudWatch drops the datapoint silently.
- `Timestamp` is `int(time.time() * 1000)` ŌĆö integer ms since epoch.
- Metric values live at the **root** of the JSON (not inside `_aws`). `_aws.CloudWatchMetrics[0].Metrics` contains only `{Name, Unit}` definitions that *reference* the root values by name.
- One EMF log line = one timestamp = up to 100 metrics (AWS hard limit). Prefer `emit_metrics()` multi-metric variant over repeated `emit_metric()` calls to cut log volume.
- Serialize with `json.dumps(..., separators=(",",":"))` to keep lines compact.

### Shared helper location
`app/services/metrics.py` is stdlib-only (no boto3, no requests, no pydantic). Any later story (US-003, US-004, US-005) should import from there ŌĆö do not re-implement the EMF schema.

### Learnings

- **EMF spec has a subtle trap**: the outer `Dimensions` array has `minItems: 1` but each inner `DimensionSet` has `minItems: 0`. That means "no dimensions" is `[[]]` NOT `[]`. Mis-reading this as `[]` produces a payload CloudWatch silently rejects (no metric ingested, no error surfaced to the producer).

- **`print(..., flush=True)` is sufficient for ECS awslogs** ŌĆö no need for `sys.stdout.write` + `sys.stdout.flush` pair, and no need for the Python logging stdlib. Simpler is better here because the helper is called from many layers.
- **`capfd` (not `capsys`) for FD-level capture**: a helper using `print(..., flush=True)` works with either, but `capfd` is the correct choice if a future helper ever calls `os.write(1, ...)` directly or if we test subprocess output. Picked `capfd` for forward-compatibility.

- **"Who served the request" vs "who was attempted"**: emitting `BackendChosen` only on the successful branch gives a clean fallback-chain donut. ### Learnings

- **Deferred post-deploy assertions pattern**: stories whose final assertion is `aws cloudwatch get-metric-statistics` on a fresh namespace cannot be settled in a PR-time tester run ŌĆö the stack-level deploy is batched. Pass the story if (a) the unit test proves the EMF line is emitted with correct namespace/dimensions/metric names, (b) the namespace literal in source matches the namespace in the post-deploy query. Record the exact deferred command in `test_report` under a "Deferred (post-deploy)" heading so whoever runs the soak test after deploy knows what to verify. - **CDK diff DependsOn churn is cosmetic when adding an IAM Policy**: attaching a new `iam.PolicyStatement` via `.task_role.add_to_policy()` generates one new `AWS::IAM::Policy` resource (`<TaskName>TaskRoleDefaultPolicy<hash>`) AND rewrites DependsOn arrays on every resource that already depended on the TaskRole. ### Learnings

- **`registerPanel(name, fn)` fires `fn` synchronously on registration** (by design ŌĆö late-joining panels still get data). That means on first page load, panels kick off a fetch before the user interacts at all. For test mocking this is slightly awkward: mocks must be installed before `setRange` runs. Workaround is to wait for `window.__monitoring__` to exist, then overwrite `fetchMetrics`/`fetchCost`, then call `refresh<Panel>(range)` to force a re-render.
- **CE GroupBy=SERVICE is summed across the range, not per-day**, so the "stacked bar" chart is an approximation ŌĆö each day's total is sliced by the across-range service mix. Honest per-day-per-service would need a second CE query with `Granularity=DAILY + GroupBy=SERVICE` and per-day service lookup. For MVP the approximation is fine; documented in code.
- **CloudWatch stat strings accept `p50`/`p95` directly** ŌĆö no need to pre-convert to `ExtendedStatistics`. `MetricDataQuery.MetricStat.Stat` takes "p50" as a string; this matched our admin API passthrough which forwards `q.stat` unchanged.

  - Warn count: `#mon-summary [data-role="warn-count"]`

### Learnings

- **Badge placement matters ŌĆö tiles have inner dots already**: the
  ingestion tile has a pre-existing `.ingest-tile__dot` (p50-driven color)
  and the upstream meter has `meter--ok|warn|danger` classes on the tile
  root. US-013 adds its own corner dot via `.badge--*::after`, placed
  top-right so it doesn't collide with existing indicators. Border-tint is
  additive ŌĆö both systems can color the same border and it degrades to the
  more-severe of the two. Worth documenting if we ever consolidate.
- **`display: none` on filtered elements still leaves classes intact**:
  the `.body-filter--crit` CSS uses `display: none` (not `visibility:
  hidden`) so DOM queries like `querySelectorAll('.badge--warn')` still
  return filtered warn tiles. A Playwright assertion like "only crit tiles
  remain visible" must check computed style / offsetParent, not a class
  filter. This is by design ŌĆö if we change filtering to class-based later,
  the counter logic (`querySelectorAll('.badge--crit').length`) still works
  because the badge class stays on the element.
- **Unit of the non-minimax share**: the fallback-chain badge compares
  `(1 - shares.minimax) * 100` (a percent) against THRESHOLDS percent
  values, while the existing yellow banner at US-010 compares
  `shares.minimax < 0.8` (a fraction). They're consistent (20% fallback ==
  minimax share 0.8), but it's easy to mis-read one as the other ŌĆö noting
  here for any future tuning.
- **`refreshAllPanels` is synchronous over async panels**: each panel's
  refresh function is `async`, but `refreshAllPanels()` fires them without
  awaiting. The immediate `recomputeSummary()` call at its tail will
  under-count badges on the first tick because fetchMetrics is still in
  flight. Each panel's own `recomputeSummary()` tail-call catches up. This
  is acceptable ŌĆö the counter converges within one poll cycle and
  mock-based tests call `refreshAll()` then `await` a microtask or use the
  per-panel `refresh<Panel>()` variants.

### Learnings

- **APP_JWT_SECRET divergence in local dev**: the .env file and AWS Secrets
  Manager each carry a DIFFERENT 64-char APP_JWT_SECRET value in this dev
  environment. The Flask subprocess (running via `from app.config import
  Config`) ended up using the AWS-sourced value because of the order
  load_dotenv() ran vs _load_secret() caching. The seeder that mints the
  JWT must either (a) explicitly pop APP_JWT_SECRET from env before
  `import Config` so Config falls back to AWS, or (b) hard-code the same
  secret the Flask will use. This caused US-009..US-012 testers to
  inadvertently skirt the problem by restoring an old token.txt. Future
  testers: always mint JWTs with `Config.APP_JWT_SECRET` and make sure
  os.environ.pop('APP_JWT_SECRET', None) runs before `from app.config
  import Config` ŌĆö matching the Flask subprocess' actual secret. This is
  now documented in US-013 validate.sh via the seeder block.
- **meter--x-api double-badge is expected, not a bug**: US-011 added
  `meter--danger|warn|ok` classes on the X API tile directly; US-013
  added `badge--crit|warn|ok` as an additive layer. When the rate-limit
  is crit, both fire at once. The tester brief explicitly allows this
  ("badge--crit OR meter--danger (both are expected markers)"). Do not
  fail a future assertion because both classes are present.
- **body-filter--crit hides BOTH warn and ok**: the CSS rule stacks
  (`.body-filter--crit .badge--ok, .body-filter--crit .badge--warn
  { display:none }`). A test that expects warn tiles to remain visible
  under crit filter will fail ŌĆö use body-filter--warn for the "warn+crit
  visible" view. This matches the intent ("show only criticals") and
  is documented in the Implementor's notes.
- **Body-aware fetch interceptor scales past per-URL mocks**: rather than
  pre-seeding `window.__mockResponses[pattern] = body`, wrap
  `SignalAuth.fetch` so that for `/admin/api/metrics/cloudwatch` it
  parses `opts.body.queries[]` and calls a user-provided
  `window.__cwResolver(query)` returning a values[] array. The mock then
  wraps into the real response shape `{series: {id: {values,
  timestamps}}}`. This lets a single interceptor serve all scenarios
  (ingestion, AI, upstream) without re-installing per test.
